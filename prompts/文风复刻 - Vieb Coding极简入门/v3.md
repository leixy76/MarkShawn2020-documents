### **01. 思维跃迁：从“聊天式AI”到“编程式AI”**

过去两年，我们与 AI 的互动，基本遵循着一种“对话”模式。

我们打开一个聊天框，像请教一位无所不知的实习生一样，对它说：“帮我写个文案”、“总结一下这篇文章”、“这个数据图表怎么做？”

这种“戳一下，动一下”的模式，对于处理原子化的、孤立的任务来说，非常高效。但一旦任务变得复杂，需要多个步骤、多个文件、多个工具协同，对话模式的短板就暴露无遗：

1.  **上下文遗忘**：聊着聊着，AI 就忘了你一开始的要求是什么。你需要一遍遍地重复背景信息，像个唠叨的甲方。
2.  **缺乏工程能力**：它无法真正地“操作”。它不能帮你读取本地文件夹里的 20 个文件，也不能帮你执行一段脚本去调用 API。它的交付物，永远是一段文本，剩下的“执行”还得靠你手动复制粘贴。
3.  **过程黑盒，结果随机**：你无法精准控制它的工作流程，也无法保证两次提问能得到同样质量的输出。每一次生成，都像一次抽卡。

而 Agentic IDE，比如你从文章开头那张截图里看到的 Codex，则彻底颠覆了这种游戏规则。

它不再是一个聊天窗口，而是一个具备“读、写、执行”能力的终端。你不再是与它“对话”，而是向它“下达指令”。你们的关系，从“用户与工具”，变成了“项目经理与程序员”。

这种转变，我称之为从“聊天式AI”到“编程式AI”的跃迁。

**“聊天式AI”的核心是“生成”。** 你给它一个 Prompt，它给你一段 Output。
**“编程式AI”的核心是“执行”。** 你给它一个**目标（Goal）**和**上下文（Context）**，它会自己**规划步骤（Plan）**、**编写代码（Code）**、**执行任务（Execute）**，并根据结果**自我修正（Self-correct）**。

你不再需要关心“如何提示”（How to prompt），你的核心工作变成了**“如何管理”**（How to manage）。你需要像一个架构师一样，思考以下问题：

*   **项目目标是什么？** (e.g., 将飞书表格的任务，无损迁移到 Linear)
*   **需要哪些原始资料？** (e.g., `tasks.csv` 文件)
*   **最终交付物应该是什么形态？** (e.g., Linear 中状态、指派、父子关系都正确的 Issues)
*   **有哪些约束和规则？** (e.g., "Antler" 相关的任务，必须归属到“公众号”父任务下)

这种思维模式的转变，才是 AI 时代最高的核心杠杆。工具本身在飞速迭代，但“架构师思维”是普适且稀缺的。

### **02. 你的第一个“数字员工”：Agentic IDE 入门**

那么，这个听起来有点硬核的 Agentic IDE 到底是什么？

别被“IDE”（集成开发环境）这个词吓到。你可以把它简单理解成一个**“能联网、会编程、能操作你电脑的命令行工具”**。

它之所以强大，因为它将大语言模型（LLM）的推理能力，与真实世界的执行能力打通了。

我直接用文章开头那个 `index.md` 里的例子，给你拆解一下一个典型的“编程式工作流”是什么样的。

这是一个真实的需求：一位主理人 @lovpen 想把从飞书导出的任务表格 `tasks.csv`，自动整理到他们的项目管理工具 Linear 里。

如果用“聊天式AI”，你可能会这么做：
1.  把 CSV 内容复制给 ChatGPT。
2.  让它帮你整理成一段段的文本。
3.  你再手动一条条复制到 Linear 里。
4.  然后手动设置截止日期、指派人员、优先级……
5.  如果有 100 条任务，这大概需要一个下午，伴随着一杯又一杯的美式咖啡和无尽的复制粘贴。

但 @lovpen 选择成为一个“Vibe 牛马”。他打开了 Codex，下达了指令：

> “这是我们从飞书导出的任务表格 tasks.csv ，请自动整理到我们的linear（https://linear.app/lovpen/inbox）/要求尽可能专业”

注意这个指令的 Vibe——**“要求尽可能专业”**。这给了 AI 巨大的发挥空间。

接下来，Codex 这个“数字员工”开始干活了：

1.  **侦察与规划（Inspect & Plan）**:
    *   它做的第一件事不是直接写代码，而是`ls -la`，查看当前文件夹里有什么。它找到了 `tasks.csv`。
    *   然后它读取了文件内容，发现表头是中文的（“任务描述”、“重要紧急程度”等）。
    *   于是它更新了计划：**“我将把这些中文表头映射到 Linear 的字段，并构建一个强大的 Python 导入脚本，这个脚本还要支持试运行（dry-run）功能。”**

2.  **编码与实现（Implement）**:
    *   它“唰唰唰”写了一个 424 行的 Python 脚本：`linear_import_from_feishu.py`。
    *   这个脚本考虑得极其周全，堪比一个资深程序员的作品：
        *   **API 交互**: 封装了与 Linear GraphQL API 的所有通信。
        *   **数据映射**: 写了 `map_priority`、`choose_state_id` 等函数，将“重要紧急”映射为 Linear 的优先级数字 `4`，将“进行中”映射为 `in_progress` 状态的 ID。
        *   **人员匹配**: 实现了 `resolve_assignee_id`，可以通过一个 `assignee_map.json` 配置文件，把飞书上的中文名（比如“南川”）精确匹配到 Linear 里的用户 ID。
        *   **鲁棒性设计**: 包含了 `--dry-run` 模式，可以在不实际创建任何数据的情况下，预览将要发生的操作。还自带了 API 请求频率控制（`--rate-limit`），防止被封。
    *   它甚至还贴心地生成了 `README.md` 和 `assignee_map.sample.json` 样例文件，告诉你这个工具怎么用。

3.  **反馈与迭代（Review & Iterate）**:
    *   第一次导入后，@lovpen 发现一个分组问题：“按照parent分组, antler这条依旧是在antler下…………”。
    *   他没有自己去改代码，而是直接把问题反馈给了 Codex。
    *   Codex 立刻理解了问题所在：“默认不破坏已有父子关系。需要带 `--reparent` 重新指定父节点。”
    *   它马上更新了另一个一键分组脚本 `linear_oneclick_grouping.py`，增加了强制重新挂载父节点的功能，并自动处理“Antler”到“公众号”的特殊逻辑。

4.  **最终执行（Execute）**:
    *   在确认 API Key 存在后，Codex 执行了最终命令。
    *   输出结果清晰明了：`Loaded 13 candidate issues.`，`Parent ensured: 公众号（2025-11）`，`Done. Parent assignments: 13`。
    *   整个过程，从一个模糊的自然语言需求，到一个专业、健壮、可复用的自动化工作流诞生，耗时不到 2 小时。而这个脚本，未来可以被整个团队复用无数次。

这就是“编程式工作”的魅力。它将一次性的、琐碎的、手动的操作，沉淀为了一个自动化的、可复用的、高质量的**“数字资产”**。

### **03. 实操指南：如何“入职”你的数字员工**

看到这里，你可能已经摩拳擦掌了。别急，要让你的数字员工高效工作，你需要像一个真正的管理者一样，为它建立一套工作规范。这套规范，可以总结为“办公室-指令-复盘”三部曲。

#### **第一步：搭建你的“办公室”（项目文件夹）**

忘掉把所有文件都堆在桌面上的坏习惯。每一个独立的工作任务，都值得拥有一个专属的项目文件夹。这是 Agent 理解你意图的基础。一个好的项目文件夹结构，就像一个干净整洁的办公室。

参考《AI写作指南3.0》的思路，我建议至少包含以下结构：

*   `/inputs` 或 `/sources`: 存放所有原始资料。比如，飞书导出的 `tasks.csv`、领导给的 `brief.docx`、网上扒下来的 `market_data.json`、十几篇参考 `report_xx.pdf`。
*   `/outputs`: 存放 Agent 生成的最终产物。比如 `final_article.md`、`import_script.py`。
*   `/style_guides`: （可选，但强烈推荐）存放你希望 Agent 模仿的风格范本。可以是你自己过去写的最好的文章、公司内部的 PPT 模板、你欣赏的公众号文章截图等。
*   `Claude.me` 或 `AGENT.md`: **这是整个项目的灵魂**。一个总指令文件，定义了 Agent 在这个项目中的**身份（Persona）**、**目标（Goal）**、**工作流程（Workflow）**和**禁止事项（Constraints）**。

一个好的 `AGENT.md` 示例：
```markdown
# 项目：2025年Q4市场分析报告

## 身份 (Persona)
你是一名资深的行业分析师，为顶级咨询公司工作。你的分析风格精准、数据驱动，同时具备前瞻性。请参考 /style_guides 文件夹中 McKinsey 的报告风格。

## 目标 (Goal)
阅读 /inputs 文件夹中的所有数据和报告，撰写一份关于 AI Agent 市场的深度分析报告，并输出为 /outputs/market_analysis_q4_2025.md。

## 工作流程 (Workflow)
1.  首先，扫描 /inputs 文件夹，列出所有可用的数据源及其核心内容摘要。
2.  提出一个详细的报告大纲，并等待我的确认。
3.  大纲确认后，逐个章节进行撰写。
4.  报告中所有引用的数据，必须明确标注来源文件。

## 禁止事项 (Constraints)
- 不要使用过于口语化或夸张的网络用语。
- 报告内容只基于提供的文件，不要从外部网络搜索未知信息。
- “Vibe 牛马”这个词只允许在引言中出现一次。
```

#### **第二步：下达你的“CEO指令”（清晰的意图）**

有了“办公室”，接下来就是下达指令。这里的核心是**提供高层级的意图，而不是微观的操作步骤。**

**错误的指令（微观管理）**:
> “打开 tasks.csv，读取第一列，然后把它作为标题，再读取第二列……”

**正确的指令（CEO 意图）**:
> “我要将飞书导出的任务（/inputs/tasks.csv）专业地迁移到 Linear。请分析文件，设计一个可靠的迁移方案并执行它。注意处理人员和状态的映射。”

把“How”交给 Agent，你只需要定义清楚“What”和“Why”。这不仅能解放你，更能激发 Agent 的“创造力”。在上面的例子中，正是因为“要求尽可能专业”这个开放性指令，Agent 才主动设计了 `--dry-run` 和 `README.md` 这些超出预期的功能。

#### **第三步：迭代与复盘（Code Review 式管理）**

没有任何一个员工能一步到位地完美完成任务，AI 也一样。关键在于建立一个高效的反馈循环。

不要因为它某个细节没做好就全盘否定，然后自己上手去改。这又回到了“聊天式”的老路。

你应该像一个 Tech Lead 在做 Code Review 一样，给出**具体、可执行**的修改意见：

*   **差的反馈**: “这个脚本不行，重写。”
*   **好的反馈**: “这个分组逻辑有问题。所有标题包含‘Antler’的任务，无论它原来的父任务是什么，都应该被强制移动到‘公众号’这个父任务下。请修改 `linear_oneclick_grouping.py` 脚本以实现这个逻辑，并增加 `--reparent` 开关。”

通过这样的方式，你不仅修正了当前的问题，更是在**“训练”**你的 Agent。它写的代码、生成的文档，在你的指导下不断完善，最终成为你真正的数字资产。

### **04. “API 一响，黄金万两”**

如果说基于本地文件的“编程式工作流”是青铜段位，那么学会利用 API，你就进入了王者领域。

API（应用程序编程接口）这个词听起来很技术，但你可以把它理解为**“让不同软件互相说话的官方语言”**。

你的 Agentic IDE 写的 Python 脚本，一旦学会了说这些“语言”，它的能力就从你的本地电脑，延伸到了整个互联网。

*   它可以连接 **Linear/Jira API**，像例子里那样自动管理你的项目。
*   它可以连接 **Notion/Airtable API**，把散乱的数据自动整理进你的知识库。
*   它可以连接 **Google Calendar API**，读取你今天的日程，然后根据会议主题，自动从 `/inputs` 文件夹里找出相关文档，并生成一个会议纪要模板。
*   它可以连接 **Twitter/Telegram API**，监控特定关键词，一旦发现重要信息，就自动进行总结，并把摘要发送到你的飞书。
*   它甚至可以连接**天气 API**，发现明天要下雨，然后自动给你在日历上创建一个“出门带伞”的提醒。

你的 Agent 不再只是一个“离线”的分析师或写手，它成了一个**全天候、跨平台的“数字总管”**。你过去需要手动在十几个 App 之间横跳、复制粘贴才能完成的工作流，现在只需要在终端里对它说一句话。

这才是 Agentic 真正的力量：**将碎片化的数字世界，为你重新整合。**

### **终章：Vibe 是你唯一的护城河**

我用 AI 写了大半年的文章，也用 Agent 自动化了许多工作流，从来没有为“AI 取代人”而焦虑过。

因为我越来越清晰地认识到，AI 正在以惊人的速度，将所有**“可被标准化的技能”**商品化（Commoditize）。

写作技巧？AI 可以模仿任何作家的风格。数据分析？AI 处理数据的速度和精度远超人类。编程？只要需求清晰，AI 写的代码比很多初级程序员都规范。

当所有执行层面的“How”都被极致地压缩成本后，什么东西会变得前所未有的珍贵？

答案是：**Vibe**。

是你的**个人风格**，你独一无二的**思考框架**，你看待世界的**独特视角**，你对“好”与“坏”的**审美判断**。

就像《AI写作指南3.0》里提到的，故事原型在几千年前就已穷尽，但《星球大战》把“哈姆雷特”的故事套上“银河帝国”的壳，就成了一个时代的文化印记。**壳（风格/Vibe）的价值，甚至超越了核（故事原型）。**

“六神磊磊读金庸”这个 IP 的价值，不在于金庸，也不在于新闻本身，而在于“六神磊磊”这个独特的视角，他建立了一个用金庸的框架来解读万物的“壳”。

在 Agent 时代，我们每个人都应该努力成为一个“建壳人”。

你的“壳”，就是你的 Vibe。它由你的经历、你的知识体系、你的品味共同构成。你把这个“壳”建立起来，然后把它变成 `AGENT.md`，变成 `/style_guides` 里的范本，把它API化。然后，让 AI 们来填充内容。

所以，未来的工作范式会变得非常清晰：

**Vibe is the new code.** 你的风格和框架，就是你下达给 AI 世界的最高指令。

**Context is the new oil.** 你能为 AI 提供多么独特、高质量、结构化的上下文，决定了它输出结果的价值上限。

**Management is the new craft.** 你驾驭、协同、迭代 AI Agent 的能力，决定了你的最终产出效率和质量。

“Vibe 牛马”不是程序员，也不是提示词工程师。他们是自己工作领域的**导演、主编和产品经理**。他们专注于定义“什么值得做”和“怎样才算好”，而将繁琐的执行过程，外包给自己麾下无数个任劳任怨、能力超群的“数字员工”。

未来的竞争，不是人和 AI 的竞争，而是**懂得如何领导 AI 的人和不懂的人**之间的竞争。

你的第一个数字员工已经整装待发，它的薪水按 Token 计算。
